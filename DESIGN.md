# 技术设计文档

## 1. 需求分析

### 1.1 背景

在日常工作和学习中，经常需要将多个短视频片段合并为一个完整视频。传统方案需要安装专业视频编辑软件（如 Premiere、剪映等），操作复杂且软件体积大。本项目旨在提供一个轻量级的浏览器端解决方案。

### 1.2 核心需求

| 需求编号 | 需求描述 | 优先级 |
|---------|---------|--------|
| R1 | 用户可上传两个本地视频文件 | 高 |
| R2 | 上传后可预览视频内容 | 高 |
| R3 | 点击合并按钮后拼接两个视频 | 高 |
| R4 | 拼接完成后可预览和下载 | 高 |
| R5 | 界面简洁美观 | 中 |
| R6 | 所有处理在本地完成 | 高 |

### 1.3 非功能需求

- **隐私安全**：视频数据不离开用户设备
- **零依赖部署**：单个 HTML 文件即可运行
- **跨平台兼容**：支持主流浏览器

## 2. 技术方案选型

### 2.1 方案对比

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| FFmpeg.wasm | 功能强大，支持多种编解码 | 需要加载 30MB+ WASM 文件，file:// 协议下受安全限制 | 不采用 |
| Canvas + MediaRecorder | 浏览器原生支持，无需外部依赖 | 输出格式仅限 WebM，需实时速率处理 | **采用** |
| MediaSource Extensions | 可直接操作媒体流 | API 复杂，兼容性差 | 不采用 |

### 2.2 最终方案

采用 **Canvas + MediaRecorder** 方案：

```
视频1 ──播放──→ Canvas ──captureStream──→ MediaRecorder ──→ Blob ──→ 下载
视频2 ──播放──→   ↑         音频流 ──────→     ↑
                  绘制帧                      合并音视频流
```

## 4. 核心算法

### 4.1 视频帧绘制算法

为了处理不同分辨率的视频，采用等比缩放 + 居中的策略：

```javascript
// 计算缩放比例，保持宽高比
scale = min(canvasWidth / videoWidth, canvasHeight / videoHeight)

// 计算绘制尺寸
drawWidth  = videoWidth × scale
drawHeight = videoHeight × scale

// 居中绘制（黑边填充）
x = (canvasWidth - drawWidth) / 2
y = (canvasHeight - drawHeight) / 2
```

### 4.2 音频处理流程

```
video 元素
    │
    ▼
createMediaElementSource()  ── 创建音频源节点
    │
    ▼
MediaStreamDestination      ── 输出到媒体流
    │
    ▼
stream.addTrack()           ── 合并到录制流
```

### 4.3 编码优先级

```
VP9 + Opus  (最优画质)
    │ 不支持
    ▼
VP8 + Opus  (兼容方案)
    │ 不支持
    ▼
默认 WebM    (最终兜底)
```

## 5. 界面设计

### 5.1 设计原则

- **简约**：去除多余元素，聚焦核心功能
- **直觉**：操作流程自上而下，符合用户心理模型
- **反馈**：每步操作都有明确的视觉反馈

### 5.2 配色方案

- 桌面端（>640px）：两个上传区域左右并排
- 移动端（≤640px）：上传区域改为上下堆叠，交换按钮旋转 90°

## 6. 测试方案

### 6.1 功能测试

| 测试项 | 测试步骤 | 预期结果 |
|--------|---------|---------|
| 文件上传 | 点击上传区域选择视频 | 显示文件名和视频预览 |

## 7. 总结

本项目通过 Canvas + MediaRecorder 方案实现了纯浏览器端的视频拼接功能，具有以下特点：

1. **零服务器依赖**：单个 HTML 文件即可运行
2. **隐私保护**：所有数据处理在本地完成
3. **技术创新**：利用现代浏览器 API 替代传统桌面软件
4. **用户友好**：简洁的 UI 设计和清晰的操作流程
